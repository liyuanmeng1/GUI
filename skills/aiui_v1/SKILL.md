# aiui_v1 — 通用企业 UI 生成 Skill

## ⚠️ 首次使用必读

> 这是一个**待配置的 UI 生成 Skill**。
>
> 在正式使用前，你需要完成三层内容的配置：
>
> - **第一层（基础层）**：声明你的前端组件库、依赖和主题配置
> - **第二层（模板层）**：提供各类页面模板（列表页/详情页/表单页/看板页）
> - **第三层（设计规范层）**：填写颜色、字体、文案、设计原则等规范
>
> **运行以下命令，立即启动开箱向导：**
>
> ```
> 帮我配置 aiui_v1
> ```
>
> 或者直接告诉 AI：**"我要开始配置 aiui_v1 skill"**

---

## 开箱状态检测（AI 执行入口）

**每次收到 UI 生成请求时，AI 必须先检查配置状态：**

```
if 01-foundation/dependencies.md 不存在 或 内容仍为占位文本:
    → 进入开箱向导（见下方「开箱引导流程」）
    → 不得直接生成代码

elif 02-templates/ 目录下无任何 .vue / .tsx 文件:
    → 提示用户先完成模板层配置
    → 引导进入「第二层：模板层配置」

else:
    → 正常执行 UI 生成工作流（见下方「工作流程」）
```

### Step A：基础组件库

**AI 提问：**

```
━━━ 第一步：基础组件库 ━━━

你们前端项目用的是哪种 UI 组件方案？

A）React + Ant Design（官方 antd 5.x）
B）Vue 3 + Element Plus（官方版）
C）使用自己维护的私有/定制组件库（有 npm 包名）
D）其他框架（请描述技术栈）
```

**收到回答后：**

- **选 A（antd）** → 追问：
  ```
  请提供以下任一信息（有什么发什么）：
  1. package.json（查看 antd 版本）
  2. 现有的 ConfigProvider 主题配置代码
  3. 或者直接告诉我：你们产品的品牌主色是什么色值？
  ```

- **选 B（Element Plus）** → 追问：
  ```
  请提供以下任一信息：
  1. package.json（查看 element-plus 版本）
  2. 现有的全局主题变量 CSS
  3. 或者直接告诉我：你们产品的品牌主色是什么色值？
  ```

- **选 C（私有组件库）** → 追问：
  ```
  请提供：
  1. 包名（如 @myorg/ui）
  2. 安装命令（如有特殊 registry）
  3. 组件命名前缀（如 Ks、My、App 等）
  4. 品牌主色色值
  ```

- **选 D（其他）** → 追问：
  ```
  请描述：
  1. 前端框架（React / Vue / Svelte / Angular？）
  2. 使用哪个 UI 组件库？
  3. 组件如何使用（import 方式、命名规则）？
  4. 品牌主色色值
  ```

**收集完毕后，AI 将信息整理写入 `01-foundation/dependencies.md`**，然后：

```
✅ 基础组件层配置完成！

━━━ 第二步：主题 Token（可选）━━━

你们有主题色 Token 配置文件吗？
- 有 → 发给我（JSON / CSS 变量 / 代码片段都可以）
- 没有 → 说"没有"，我根据主色帮你生成一套基础 Token
```

收到后写入 `01-foundation/theme.json`（或跳过），进入下一步。

## 工作流程（配置完成后，收到 PRD 时执行）

```
Step 0  读取全局框架文件，确认 AppLayout 结构
   ↓
Step 1  分析 PRD → 列出页面清单 + 追问缺失信息
   ↓
Step 2  并行 read_file 读取：
           ① 涉及页面的所有模板文件（02-templates/）
           ② 03-design-db/ 中相关规范文件
   ↓       ← 未完成此步，不得执行任何代码生成
Step 3  制定逐页改造计划（保留 / 修改 / 新增 / 删除）
   ↓
Step 4  以模板为基础生成代码
   ↓
Step 5  对照自检清单逐条核查后才算交付完成
```

### Step 1：分析 PRD

提取：
1. 涉及哪几个页面？对应哪种模板？
2. 页面间跳转关系
3. 每个页面的核心字段、操作、状态

信息不足时，**主动追问以下项**（一次性追问，不分多轮）：

| 维度 | 追问问题 |
|------|---------|
| 操作类型 | 是新建还是编辑？还是都有？ |
| 分页 | 列表是否需要分页？默认每页几条？ |
| 状态流转 | 资源有哪些状态？哪些状态可以操作？ |
| 批量操作 | 是否需要批量操作？ |
| 数据来源 | 下拉选项是静态枚举还是接口动态获取？ |
| 空状态 | 首次使用是否有引导？ |

### Step 2：读取模板（强制，不可跳过）

根据 Step 1 的页面清单，**立即并行 read_file** 读取：

| 页面类型 | 读取路径 |
|---------|---------|
| 全局框架 | `{skill_dir}/02-templates/AppLayout/` |
| 列表页 | `{skill_dir}/02-templates/ListPage/` |
| 详情页 | `{skill_dir}/02-templates/DetailPage/` |
| 表单页 | `{skill_dir}/02-templates/FormPage/` |
| 看板页 | `{skill_dir}/02-templates/DashboardPage/` |
| 组件规范 | `{skill_dir}/03-design-db/component-spec.md`（若存在） |

> ⛔ **未读取模板，不得生成代码。** 模板的 props 接口、CSS 类名、层级结构只存在于模板源文件中。

### Step 3：制定改造计划

对模板中每个功能区域做决策：

| 决策 | 含义 |
|------|------|
| ✅ 保留 | PRD 中有，直接复用模板结构 |
| ✏️ 修改 | PRD 中有但内容不同，替换业务字段 |
| ➕ 新增 | PRD 有但模板没有，在副本中添加 |
| ❌ 删除 | PRD 没有，从副本中移除 |

### Step 4：生成代码

1. 完整复制模板到 `src/pages/{PageName}/`
2. 按 Step 3 计划逐区域修改副本（不动原模板）
3. 颜色/间距使用 `01-foundation/` 中定义的 Token / CSS 变量
4. 文案遵循 `03-design-db/copywriting.md`

### Step 5：交付前自检

| # | 检查项 | 要求 |
|---|--------|------|
| 1 | Step 2 模板已读取 | 本次 read_file 记录中包含所有涉及模板路径 |
| 2 | 页面嵌套在 AppLayout 内 | ✅ |
| 3 | 无硬编码颜色 | 全部使用 Token / CSS 变量 |
| 4 | 空值处理 | 统一显示 `–`（参考 copywriting.md） |
| 5 | 操作列固定 | 表格操作列 `fixed: 'right'` |
| 6 | 禁止自造组件库已有组件 | 直接使用组件库原生组件 |
| 7 | Modal 关闭销毁 | 使用组件库对应 API |

## ⛔ 模板保护规则

**`02-templates/` 目录下的所有文件为只读模板，任何情况下不得直接修改。**

- 开发新页面时，**完整复制**模板到 `src/pages/{PageName}/` 目录下
- 在副本上修改，原模板保持不变
- 只有用户明确说「更新模板」时，才允许修改 `02-templates/` 内的文件

## 三层架构总览

```
01-foundation/          ← 第一层：基础层（依赖声明 + 主题 Token）
02-templates/           ← 第二层：模板层（可复用页面模板）
03-design-db/           ← 第三层：设计规范层（颜色/字体/文案/原则）
src/pages/              ← 业务层（已交付的业务页，不在 skill 内）
```
